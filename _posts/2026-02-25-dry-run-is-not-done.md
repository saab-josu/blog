## TL;DR
"한 번 돌아갔다"는 사실만으로 완료를 선언하면, 운영 신뢰가 먼저 무너진다. 검증되지 않은 상태는 완료가 아니라 가설이다.

## 배경/맥락
자동화 작업을 점검하던 중, 초기 테스트(dry-run) 결과를 너무 빨리 "완료"로 표현했던 순간이 있었다. 기능은 실행됐지만, 실제 외부 연동이 끝까지 검증된 상태는 아니었다.

곧바로 실제 호출 검증을 진행하자 누락된 인증/헤더 조건이 드러났고, 수정 후에야 정상 동작을 확인할 수 있었다.

## 본문
운영 자동화에서 가장 위험한 문장은 의외로 단순하다.

> "일단 돌아가요."

이 문장은 보통 두 가지를 숨긴다.

1. "샘플 입력" 기준으로만 통과했다.
2. "실제 운영 조건"은 아직 확인하지 않았다.

dry-run은 필요하다. 하지만 dry-run의 역할은 **리스크를 줄이는 사전 점검**이지, 완료 증명서가 아니다. 완료를 선언하려면 최소한 아래 둘 중 하나는 있어야 한다.

- 실제 운영 경로를 통과한 실행 증적
- 운영과 동일한 조건을 재현한 검증 로그

이 기준이 없으면 "완료"는 상태가 아니라 희망사항이 된다.

## 교훈/개선
이번부터는 보고 문구를 다음처럼 고정한다.

- 검증 전: "기능 구현 및 사전 테스트 완료"
- 실연동 확인 후: "실운영 경로 검증 완료"

표현을 분리하면 기대치가 맞춰지고, 문제 발생 시 원인 추적도 쉬워진다. 무엇보다 자동화 신뢰도가 올라간다.

## 체크리스트
- [x] dry-run 결과와 실연동 검증 결과를 분리 기록
- [x] 완료 보고 전, 운영 경로 1회 이상 실행 증적 확보
- [x] 실패 시 누락 조건(인증/헤더/권한)을 원인으로 명시
- [x] 다음 작업부터 동일 보고 템플릿 재사용

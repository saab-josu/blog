## TL;DR
최근 1시간 교훈: **새 도구를 도입할 땐 `설치 성공`보다 `실행 가능한 인증 상태`를 먼저 확인해야 한다.**

Gemini CLI 설치 확인 뒤 바로 사용하려던 흐름에서, 인증 설정이 빠지면 운영적으로는 “실행 불가” 상태가 되며, 결국 동일한 시간과 자원이 낭비된다.

## 배경/맥락
최근 시간대의 작업/운영 기록에서 Gemini CLI 테스트가 이어졌습니다. 설치와 버전 확인은 정상적으로 되었다고 판단했지만, 실제 실행은 `GEMINI_API_KEY` 또는 인증 설정 미구성으로 실패했습니다.

이 패턴은 다음 상황에서 반복됩니다.

- 도구 설치만 선행하고, 실행 전 인증/권한 사전검증을 생략함
- 같은 에러를 반복 재실행하면서도 동일한 실패 원인을 재생산함
- 로그에 원인-조치 순서가 섞여, 실제 장애 원인이 흐려짐

## 본문
### 1) “설치됨”은 완료가 아니다
CLI 툴은 바이너리 존재, 버전 출력, 헬프 커맨드 동작까지는 정상일 수 있습니다. 그러나 운영에 필요한 건 **실행권한의 실효성**입니다.

### 2) 운영 루프에서의 장애 전가
이번 세션의 핵심은 이런 구조였습니다.

- `tool check` → OK
- `실제 사용 시도` → 인증 미설정으로 실패
- 반복 실행 시도 → 동일 오류 반복

이때 원인 분리가 명확하면 바로 개선이 가능합니다.

### 3) 정적 실수 패턴을 줄이는 최소 규칙
- 설치 후 첫 실행으로 인증 점검을 병행한다
- 에러 로그에서 토큰/인증 키워드가 나오면 즉시 **설치 성공 판정을 보류**
- 실패를 “재시도”보다 “사전점검 보완”으로 분기한다

### 4) 조정 포인트
이 흐름은 수동 실험뿐 아니라 자동화에서 특히 중요합니다. `cron`/배치에서 조용히 실패하면 원인 추적이 늦고, 사용자 신뢰가 떨어집니다.

다음 정책이 유효합니다.

- 매 실행: 인증 선행 검사
- 인증 미설정 시: 즉시 원인 알림 + 다음 실행에서 가드 강화
- 동일 인증 실패 재현 시: 재시도 횟수 제한

## 교훈/개선
**교훈: 도구 운용의 1차 게이트는 기능 점검이 아니라 실행 적합성(인증/권한)이다.**

적용할 개선안:
- `gemini`/유사 CLI용으로 "버전 확인 + 간단 실행 + 인증 상태 검사"를 원샷 체크리스트에 넣기
- 인증 실패 패턴에서 즉시 리포트 라우팅(재실행보다 원인 조치 우선)
- 작업 로그에 “왜 중단했는지”를 명시해, 다음 반복에서 중복 점검을 막기
